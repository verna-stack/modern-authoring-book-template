Automation Workflows {#sec-workflows-deepdive}

Computational manuscripts are unique because they blend static prose with dynamic code execution. While running live Python or R cells inside your chapter documents guarantees that your figures and datasets are always current, executing those blocks on every single compile creates a dangerous layout processing bottleneck.

To bypass this bottleneck, we implement the Computational Freeze Pattern at the root of our compilation parameters.

The Architecture of the Freeze Parameter

Adding the freeze: true property to our global configuration space changes how the Quarto compiler handles code cells:

execute: freeze: true

Under normal compilation behaviors, running quarto render forces the system to execute every single code block in the book from scratch. If you have fifty chapters containing data science calculations, compiling a simple spelling fix in Chapter 1 requires waiting for all fifty chapters to completely re-evaluate their code.

The freeze: true directive overrides this loop by enforcing three strict execution parameters:

The Write-Through Phase: When you edit a .qmd file and run quarto render, Quarto detects that the source file has changed. It runs your Python virtual environment sandbox, executes the code blocks, and saves the resulting data (such as code outputs, text console prints, and charts) as localized JSON assets inside a hidden directory named _freeze/ at your project root.

The Read-Only Phase: If you render the book again without changing the computational code in that specific file, Quarto bypasses the Python interpreter entirely. It reads the pre-compiled layout blocks straight out of the _freeze/ ledger and instantly stitches them into your HTML or PDF print assets.

The CI/CD Decoupling Loop: Because the execution results are frozen into plaintext JSON payloads, we can commit the _freeze/ directory directly to our Git repository. When your GitHub Actions server compiles your book, it does not need Python, Matplotlib, or any local data drivers installed. It simply reads your frozen assets and builds your clean web book instantly.

Freeze Configuration Mappings

You can tune the freeze environment by adjusting the parameters in your _quarto.yml file to target three distinct workflows:

freeze: true: (Highly Recommended for CI/CD) Code blocks are only executed when their source files are explicitly edited during local development. The cloud pipeline relies entirely on the frozen cache.

freeze: auto: Code blocks are automatically re-executed if their source code has changed, but are skipped if they match the active layout cache.

freeze: false: (Default) Code blocks are always executed on every render pass.

The Version Control Rule for Frozen Assets

Because the _freeze/ directory acts as your computational pipeline’s database, you must follow a strict version control protocol:

Do NOT add _freeze/ to your .gitignore file. If your cloud runner cannot find the frozen JSON assets, it will attempt to run the Python code blocks itself. Because the cloud runner lacks your isolated PyCharm virtual environment keys and local variables, your build pipeline will crash.

Always commit changes to _freeze/ alongside your code modifications. When you alter a Python graphing loop, run a local render to update the frozen assets, stage both the .qmd and the _freeze/ paths, and push them to GitHub together:

git add introduction.qmd _freeze/ git commit -m “chore: update telemetry plotting parameters and freeze execution states”

Troubleshooting Computational Reference Drift

When combining the computational freeze engine with cross-referenced assets, three critical pipeline exceptions can occur. Understanding their mechanics prevents compiler warnings during local and cloud builds.

  1. The Cache-Bypass Indexing Failure

By default, Quarto’s background processor attempts to optimize rendering times by aggressively caching executed Jupyter notebook results. However, this caching mechanism can occasionally “blind” Quarto’s global indexing engine.

If the compile pipeline pulls a pre-rendered figure straight from the cache, it may skip registering its identification handle (e.g., Figure 3.1) in the global cross-reference ledger. This results in the compiler outputting a warning: Unable to resolve crossref.

The Resolution Pattern: For any dynamic code chunk that serves as a target for a cross-reference link, explicitly bypass the execution cache by appending the #| cache: false directive directly to the cell options:

#| label: fig-pipeline-metrics #| fig-cap: “Pipeline Compilation Latency Scaling Curve” #| echo: false #| cache: false

  1. The Preface Numbering Collision

In traditional book layout structures, your index page (index.qmd) serves as unnumbered frontmatter (the Preface). Because this page lacks a formal chapter integer, generating active, numbered figures inside it creates a mathematical reference collision.

If you attempt to link to a figure located inside unnumbered frontmatter from a formal, numbered chapter (like Chapter 1 or Chapter 2), the LaTeX engine cannot resolve the relative numbering offset and will drop a broken link warning [?].

The Resolution Pattern: Always isolate computational code blocks and their parent headers within formal, numbered chapter files (e.g., introduction.qmd). Keep your index.qmd file strictly limited to static introductory prose, steering clear of any dynamic code execution blocks.

  1. The Functional Execution Mandate

A highly subtle, engine-specific optimization occurs within Quarto’s underlying dependency-tracking graph. If the Python script inside your cell merely assigns static, hardcoded dummy arrays (e.g., raw literal coordinates with zero active evaluation logic) and immediately renders a plot, the internal Jupyter parsing sequence may optimize its evaluation pipeline footprint. Under specific configuration parameters, this optimization can cause the engine to bypass full metadata registration, rendering the visual element on your page but failing to bind its identifier handle to the global cross-reference index.

The Resolution Pattern: To guarantee that your asset remains registered in the compiler’s tracking database, ensure your Python code block performs active mathematical calculations, models dynamic curves, or evaluates functional equations rather than relying on inert dummy values. For example, instead of feeding a completely static list directly to your plotter, programmatically generate the coordinates using a functional mathematical relationship, such as calculating values along a function line \(y = f(x)\):

#| label: fig-pipeline-metrics #| fig-cap: “Pipeline Compilation Latency Scaling Curve” #| echo: false #| cache: false

import matplotlib.pyplot as plt

steps = list(range(1, 6)) latency = [x * 10 for x in steps] # Linear computational mapping function f(x) = 10x
